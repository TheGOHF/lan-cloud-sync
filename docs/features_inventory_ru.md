# LAN Cloud Sync: Инвентаризация возможностей

Документ составлен только по фактам, видимым в текущем репозитории. Он не включает абстрактные советы и не приписывает проекту функций, которых нет в коде.

## 1. Уже реализованные возможности системы

### Возможности на стороне сервера

- В `server/app/main.py` реализовано FastAPI server application.
- Реализован HTTP API для получения списка метаданных файлов:
  `GET /files`.
- Реализован HTTP API для загрузки файла:
  `POST /upload`.
- Реализован HTTP API для скачивания файла:
  `GET /download`.
- Реализован HTTP API для удаления файла:
  `DELETE /files`.
- Сервер хранит метаданные файлов в SQLite `server/data.db`.
- Сервер хранит физические файлы в `server/storage/`.
- Сервер вычисляет SHA-256 для загруженных файлов.
- Сервер увеличивает версию файла при изменении содержимого.
- Сервер увеличивает версию файла при удалении.
- Сервер хранит состояние удаления через флаг `deleted`.
- Сервер хранит `device_id` стороны, внесшей последнее изменение.
- Сервер валидирует относительные пути и отклоняет абсолютные пути и `..`.
- Сервер отдаёт файлы потоково по чанкам.
- Сервер удаляет физический файл из storage при удалении.
- Сервер согласует метаданные в БД с фактическим содержимым storage при вызове `GET /files`.
  Это включает:
  создание metadata rows для файлов в storage, которых нет в БД,
  пометку DB rows как deleted при отсутствии физических файлов,
  восстановление удалённых DB rows, если соответствующие файлы снова есть в storage.

### Возможности на стороне клиента

- В `client/app/cli/main.py` реализован CLI client.
- Реализованы CLI-команды:
  `sync`, `status`, `upload`, `download`, `list`, `watch`.
- Клиент хранит локальное состояние синхронизации в SQLite `client/data/sync_state.db`.
- Клиент рекурсивно сканирует локальную папку синхронизации.
- Клиент вычисляет SHA-256 для локальных файлов.
- Клиент получает `mtime` файла при локальном сканировании.
- Клиент запрашивает удалённые метаданные с сервера.
- Клиент загружает локальные файлы на сервер.
- Клиент скачивает удалённые файлы с сервера.
- Клиент отправляет delete requests на сервер.
- Клиент строит sync plan сравнением:
  локального состояния файловой системы,
  удалённых метаданных,
  локального состояния из БД.
- Клиент поддерживает следующие sync actions:
  `upload`,
  `download`,
  `delete_remote`,
  `delete_local`,
  `mark_local_deleted`,
  `conflict_download`.
- Клиент сохраняет локальное tombstone state с `deleted=True`.
- Клиент обрабатывает конфликт так:
  сохраняет локальную версию как отдельный `_conflict` файл,
  затем скачивает удалённую версию в исходный путь.
- Клиент выполняет начальный sync cycle при старте watcher.
- Клиент следит за изменениями локальной файловой системы через `watchdog`.
- Клиент делает debounce локальных file events через `threading.Timer`.
- Клиент периодически опрашивает сервер в фоне потоке.
- Клиент предотвращает перекрывающиеся sync cycles через `Lock`.

### Операционные и вспомогательные возможности

- В `scripts/` есть start scripts для сервера и client watcher.
- В `scripts/` есть stop scripts для сервера и client watcher.
- В `scripts/check_status.bat` есть status-check script.
- Логи пишутся через script-level stdout/stderr redirection в `logs/`.

## 3. Нужен ли минимальный GUI для защиты

### Оценка по фактам

- Текущий репозиторий уже предоставляет рабочую поверхность взаимодействия через CLI-команды и start/stop/status scripts.
- В текущем репозитории нет никакой GUI-реализации.
- Поведение системы, которое можно продемонстрировать по коду, включает:
  запуск сервера,
  запуск client watcher,
  обнаружение локальных изменений файлов,
  upload/download/delete поведение,
  создание conflict copy,
  вывод локального sync-state состояния.

### Вывод только по репозиторию

- Минимальный GUI технически не нужен для запуска системы, потому что реализованный интерфейс уже CLI-based, а operational scripts уже есть.
- Минимальный GUI был бы дополнительной функцией, а не уже реализованной частью системы.
- Если формат защиты требует визуального взаимодействия, текущий репозиторий такого слоя не предоставляет.

### Чего именно не хватает для GUI support

- Нет GUI entry point.
- Нет использования GUI framework.
- Нет view layer для списка файлов, статуса синхронизации, conflict state или состояния подключения к серверу.
- Нет event wiring от sync engine к каким-либо visual components.
