# LAN Cloud Sync: Структура и архитектура проекта

Документ составлен только по фактам, видимым в текущем репозитории. Если вывод не доказан напрямую кодом, он явно помечается как предположение.

## 1. Точки входа проекта

### Сервер

- Основная точка входа приложения: `server/app/main.py`.
- Этот модуль создаёт `FastAPI`-приложение, вызывает `init_db()` на старте и подключает роутер API.
- В репозитории есть рабочий скрипт запуска: `scripts/start_server.bat`.
- Этот скрипт запускает сервер так:

```bat
python -m uvicorn server.app.main:app --host 0.0.0.0 --port 8000
```

### Клиент

- Реальная точка входа клиента: `client/app/cli/main.py`.
- Этот модуль реализует CLI-приложение `lan-cloud-sync` на основе `argparse`.
- Поддерживаемые команды: `sync`, `status`, `upload`, `download`, `list`, `watch`.
- В репозитории есть рабочий скрипт запуска фонового клиента: `scripts/start_client_watch.bat`.
- Этот скрипт запускает клиент так:

```bat
python -m app.cli.main watch --device-id <device_id>
```

### Другие наблюдения

- `client/app/main.py` содержит только docstring и не выглядит как активная точка входа.
- В каталоге `scripts/` также есть вспомогательные скрипты запуска, остановки и проверки статуса.

## 2. Где находятся серверная и клиентская части

### Серверная часть

- Корневой каталог: `server/`.
- HTTP API: `server/app/api/routes.py`.
- Слой базы данных: `server/app/db/`.
- Модель метаданных: `server/app/models/file.py`.
- Серверная логика работы с файлами и метаданными: `server/app/services/file_service.py`.
- Логика физического хранения файлов: `server/app/services/storage_service.py`.

### Клиентская часть

- Корневой каталог: `client/`.
- CLI: `client/app/cli/main.py`.
- Сетевое взаимодействие: `client/app/sync/network.py`.
- Планирование синхронизации и выполнение действий: `client/app/sync/sync_engine.py`.
- Локальная база метаданных: `client/app/sync/db.py`.
- Сканирование локальных файлов и хеширование: `client/app/sync/file_utils.py`.
- File watcher и цикл опроса: `client/app/sync/watcher.py`.

### Общая часть

- Корневой каталог: `shared/`.
- Общие Pydantic-схемы ответов API: `shared/schemas.py`.

## 3. Основные модули и их назначение

### Серверные модули

- `server/app/main.py`
  Создаёт FastAPI-приложение, инициализирует БД, подключает маршруты.
- `server/app/api/routes.py`
  Определяет HTTP endpoints: `GET /files`, `POST /upload`, `GET /download`, `DELETE /files`.
- `server/app/db/session.py`
  Настраивает SQLite engine, фабрику сессий, `get_db()` и `init_db()`.
- `server/app/db/base.py`
  Объявляет базовый класс SQLAlchemy.
- `server/app/models/file.py`
  Определяет таблицу `files` с полями `path`, `version`, `hash`, `updated_at`, `device_id`, `deleted`.
- `server/app/services/file_service.py`
  Реализует создание и обновление метаданных, soft delete, сериализацию метаданных и синхронизацию состояния БД с физическим storage.
- `server/app/services/storage_service.py`
  Реализует нормализацию путей, сохранение upload, listing файлов, чтение по чанкам и физическое удаление из storage.
- `server/app/services/hashing.py`
  Вычисляет SHA-256 файлов.

### Клиентские модули

- `client/app/cli/main.py`
  CLI-оболочка над функциями синхронизации.
- `client/app/sync/config.py`
  Хранит клиентские константы: путь к папке синхронизации, путь к локальной БД, URL сервера, размер чанка, интервалы polling/debounce.
- `client/app/sync/db.py`
  Хранит локальное состояние синхронизации в таблице SQLite `local_files`.
- `client/app/sync/file_utils.py`
  Сканирует локальную папку, читает файлы чанками, считает SHA-256, получает `mtime`.
- `client/app/sync/network.py`
  Реализует HTTP-клиент на `requests` для операций list/upload/download/delete.
- `client/app/sync/sync_engine.py`
  Центральная sync-логика: строит план синхронизации, определяет конфликты, применяет upload/download/delete и ведёт локальные tombstone.
- `client/app/sync/watcher.py`
  Интегрирует `watchdog`, делает debounce локальных событий через `Timer` и опрашивает сервер в отдельном фоне потоке.

## 5. Как устроено сетевое взаимодействие

### Протокол и адресация

- Клиент использует HTTP через `requests`.
- Базовый URL клиента по умолчанию: `http://127.0.0.1:8000` в `client/app/sync/config.py`.
- Скрипт запуска сервера поднимает Uvicorn на `0.0.0.0:8000`.
- Факт из репозитория:
  сервер настроен слушать все интерфейсы,
  но клиентская конфигурация по умолчанию всё ещё указывает на localhost.

### HTTP API

- `GET /files`
  Возвращает список метаданных файлов.
- `POST /upload`
  Принимает файл, а также `path` и `device_id` через `multipart/form-data`.
- `GET /download?path=...`
  Потоково возвращает бинарное содержимое файла клиенту.
- `DELETE /files?path=...&device_id=...`
  Помечает файл удалённым и удаляет его из серверного storage.

### Детали передачи данных

- Upload идёт потоково как вручную сформированные multipart-данные через `MultipartUploadStream`.
- Download идёт чанками по `1 MiB`.
- Ответы сервера на скачивание тоже отдаются чанками через `iter_file_chunks`.
- Сетевые ошибки на клиенте оборачиваются в `NetworkError`.

### Наблюдение по delta sync

- `client/app/sync/network.py` поддерживает опциональный параметр `updated_since` в `get_files()`.
- Серверный маршрут `GET /files` этот параметр не принимает.
- В `server/app/services/file_service.py` есть TODO про добавление pagination и filters `updated_since`.
- В `client/app/sync/sync_engine.py` есть TODO про повторное включение delta sync после поддержки на сервере.
- Следовательно, по текущему коду синхронизация метаданных основана на полном списке, а не на delta-механизме.

## 6. Как реализована синхронизация файлов

### Общая схема

- Клиент строит три представления состояния:
  локальное состояние файловой системы из `scan_local_folder()`,
  удалённое состояние сервера из `GET /files`,
  локальную БД состояния из `local_files`.
- `build_sync_plan()` сравнивает эти представления по `path`, `hash`, `version`, `deleted` и иногда `mtime`.
- Результат — список объектов `SyncAction`.

### Поддерживаемые действия

- `upload`
  Отправить локальный файл на сервер.
- `download`
  Скачать серверный файл в локальное хранилище.
- `delete_remote`
  Отправить запрос на удаление на сервер.
- `delete_local`
  Удалить локальный файл, потому что сервер пометил его удалённым.
- `mark_local_deleted`
  Оставить tombstone только в локальной БД.
- `conflict_download`
  Сохранить текущий локальный файл как `<name>_conflict<suffix>`, затем скачать серверную версию.

### Поведение при разрешении состояний

- Если файл есть локально, но нет на сервере, клиент планирует `upload`.
- Если файл есть на сервере, но отсутствует локально:
  если история в локальной БД показывает, что он раньше существовал и не был удалён, клиент планирует `delete_remote`;
  иначе он планирует `download`.
- Если сервер помечает файл удалённым:
  клиент либо удаляет локальный файл, либо записывает локальный tombstone, либо повторно загружает файл, если он был создан локально заново.
- Если и локальное, и удалённое состояние изменились относительно последнего снимка в локальной БД, клиент создаёт conflict action.

### Версионирование

- У серверных метаданных есть целочисленное поле `version`.
- Версия увеличивается при изменении содержимого или удалении.
- Клиент хранит последнюю известную версию в локальной БД и использует её при сравнении.

### Обнаружение изменений

- И сервер, и клиент используют SHA-256.
- Клиент также использует локальный `mtime` в одном конкретном случае:
  когда локальный и удалённый контент различаются, но в локальной БД ещё нет предыдущей записи для этого файла.
  В этом случае выбор между upload и download делается сравнением локального `mtime` и удалённого `updated_at`.

### Хранение файлов

- Физические серверные файлы хранятся в `server/storage/`.
- Сервер строит пути хранения из нормализованных относительных путей.
- Нормализация пути запрещает абсолютные пути и `..`.

## 7. Хранение метаданных, логирование, конфигурация, многопоточность

### Хранение метаданных

Да, есть два отдельных хранилища метаданных.

- Серверная БД: `server/data.db` (SQLite).
  Таблица `files` хранит:
  `id`, `path`, `version`, `hash`, `updated_at`, `device_id`, `deleted`.
- Клиентская БД: `client/data/sync_state.db` (SQLite).
  Таблица `local_files` хранит:
  `path`, `hash`, `version`, `last_synced`, `conflict`, `deleted`.

### Логирование

- Код использует Python `logging`.
- Явный `logging.basicConfig(...)` настроен в `client/app/cli/main.py`.
- `watcher.py` и `sync_engine.py` используют `logger.info`, `logger.warning`, `logger.debug` и `logger.exception`.
- На серверной стороне явный `logging.basicConfig(...)` не найден.
- В репозитории есть операционные log-файлы:
  `logs/server.log`, `logs/client-watch.log`, `logs/check-status.log`.
- Эти log-файлы создаются bat-скриптами через перенаправление stdout/stderr.
- Отдельный журнал операций в БД или специализированный event log module не найден.

### Конфигурация

- Runtime-конфигурация клиента хранится как константы в `client/app/sync/config.py`.
- Основные значения:
  `BASE_PATH`, `LOCAL_DB_PATH`, `SERVER_URL`, `CHUNK_SIZE`, `POLL_INTERVAL_SECONDS`, `LOCAL_EVENT_DEBOUNCE_SECONDS`.
- Серверные пути к БД и storage также захардкожены в коде:
  `server/app/db/session.py` и `server/app/services/storage_service.py`.
- Отдельный `.env`, YAML, JSON или выделенный settings layer в репозитории не найден.

### Многопоточность

Да, клиент явно использует многопоточность.

- `watchdog.Observer` запускает наблюдение за файловой системой.
- `threading.Timer` используется для debounce локальных событий.
- Daemon `Thread` используется для периодического remote polling.
- `Lock` предотвращает перекрывающиеся sync cycles.
- `Event` используется для остановки polling loop.

### Замечание по ограничению

- Обе конфигурации SQLite engine используют `check_same_thread=False`.
- Это прямой факт из кода.
- Более сильное утверждение о полной потокобезопасности всего приложения было бы предположением.

## 8. Общая архитектурная картина

- Репозиторий разделён на клиент, сервер и общие схемы.
- Сервер предоставляет небольшой HTTP API для list, upload, download и delete операций.
- Сервер хранит:
  физическое содержимое файлов в `server/storage/`,
  метаданные и версии в SQLite `server/data.db`.
- Клиент хранит:
  синхронизируемые рабочие файлы под `BASE_PATH`,
  собственное sync-state состояние в SQLite `client/data/sync_state.db`.
- Синхронизация реализована как комбинация:
  наблюдения за локальными файлами,
  периодического опроса сервера,
  сравнения локальной файловой системы, списка серверных метаданных и локальной sync-state БД.
- Конфликты обрабатываются сохранением локальной версии в отдельный `_conflict` файл, после чего скачивается удалённая версия.
- Удаления представлены как soft-delete в метаданных и физическое удаление из серверного storage.
- По текущему коду настоящий delta-sync API ещё не реализован.

## Что помечено как предположение

- Разумно описывать проект как LAN client-server file sync system, потому что:
  сервер запускается на `0.0.0.0:8000`,
  а название и структура репозитория указывают на это.
  Однако клиентский `SERVER_URL` по умолчанию всё ещё равен `127.0.0.1`, поэтому для реального multi-host LAN-использования потребуется изменение конфигурации.
- В коде не найдены server-side background job queue, message broker, WebSocket channel или push-based sync mechanism.
  Это надёжное отрицательное наблюдение о текущем репозитории, а не утверждение о будущих планах.
